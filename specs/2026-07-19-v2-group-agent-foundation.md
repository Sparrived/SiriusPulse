# V2 群聊完整 Agent：复盘与第一阶段基线

## 目标

V2 将当前“模型回复后串行执行工具”的半 Agent，演进为面向多人群聊的完整 Agent。核心不是让模型更频繁调用工具，而是让每一轮都具有可审计的决策、执行、确认和记忆边界。

目标状态：

```text
接收群消息 -> 判断是否响应/响应对象 -> 形成可见回复或任务目标
-> 选择最小工具集 -> 执行并校验结果 -> 基于结果回复 -> 写入可用记忆
```

普通闲聊可在“判断是否响应”后直接结束；复杂任务进入隐藏计划流程。工具调用、状态变更和最终答复必须是三个可分离的阶段。

## 当前架构复盘

现有系统已有可复用基础：`DelayedResponseQueue` 聚合群消息，`ContextAssembler` 提供历史和记忆，`Brain` 负责模型调用，`SkillRegistry/SkillExecutor` 负责工具，并已有计划模式和公开进度。问题集中在边界不一致：

- 同一工具在同步和异步路径的参数、默认值、类型和权限校验曾不一致。
- 延迟队列为所有工具无差别重试，可能重复产生副作用；超时配置也未传入执行器。
- 工具原始输出直接进入下一轮模型上下文，既无长度上限也无“数据不是指令”的边界。
- 人格字段没有完整进入 prompt；自定义 system prompt 会绕过基础运行约束；身份和“多用互动工具”的指令被放入用户消息，破坏消息角色边界。
- 所有可见工具会一次性暴露给模型，多个工具调用按顺序执行，但没有依赖计划、结果校验或失败后的重规划。

## 研究依据

- Toolformer、API-Bank、Gorilla、ToolLLM 说明工具调用的关键不只是能否生成函数名，还包括候选工具选择、参数正确性和受限调用环境。
- RestGPT 将复杂 API 使用拆为计划、调用与结果解析，适合 V2 的 `plan -> act -> verify` 分层。
- ToolSandbox、tau-bench 强调有状态环境、终态、策略约束和确认条件；群聊中的禁言、撤回、外发正属于这类任务。
- ComplexFuncBench 与工具鲁棒性研究显示，长上下文和相似工具扩张会明显降低可靠性，因此 V2 需要按任务检索小工具集，而非全量注入 schema。
- RoleLLM、Beyond Dialogue 说明人格/角色必须与具体场景和对话行为一致，支持把人格固定在 system 层、把群消息和动态资料保留在用户上下文层。
- HuixiangDou 与 AutoGen 分别提供群聊拒答/检索和受控发言者调度的参考，支持将“是否该插话”与“如何完成任务”分开评估。

参考：Toolformer (arXiv:2302.04761)、API-Bank (arXiv:2304.08244)、Gorilla (arXiv:2305.15334)、RestGPT (arXiv:2306.06624)、ToolLLM (arXiv:2307.16789)、AutoGen (arXiv:2308.08155)、RoleLLM (arXiv:2310.00746)、HuixiangDou (arXiv:2401.08772)、tau-bench (arXiv:2406.12045)、ToolSandbox (arXiv:2408.04682)、ComplexFuncBench (arXiv:2501.10132)。

## 第一阶段：工具与提示词边界

- `SkillExecutor` 对同步和异步工具共用参数准备、开发者权限、管理员资格、默认值、类型转换与运行时依赖注入。
- `SkillDefinition` 新增 `retry_safe`。默认不重试；队列只为显式标记的只读工具重试，并将 `skill_execution_timeout` 传入执行器。`web_lookup` 是当前首个标记实例。
- `SkillResult.to_model_text()` 以固定边界、最大长度和引用数据说明注入工具结果；群聊队列不再注入原始执行结果。
- 人格 prompt 覆盖动机、情绪范围、常用开场/收尾；自定义人格保留不可覆盖的工具与身份约束。
- 工具使用规则改为任务/状态驱动，取消“为了群聊互动而主动调用”和“每轮必须 stop”；动态记忆和画像以参考上下文标识注入。

这阶段故意不引入新的大框架或第二个调度器，先统一现有运行边界并以回归测试固定契约。

## 第二阶段：Agent 回合与工具检索

- `AgentTurn` 已把 `observe -> decide -> plan -> act -> verify -> respond -> complete` 作为每个延迟群聊回合的显式事件轨迹；事件只含工具名、状态和限长摘要，不记录完整参数或结果。
- `Brain` 可通过 `skill_query` 和 `max_skill_candidates` 请求 `SkillRegistry.build_tools_for_query()`，由词法匹配在权限、平台和管理员过滤后给模型注入有上限的候选 schema。
- 队列会从整批 `<message>` 中提取正文作为检索查询，不把发言者 XML 属性混入工具选择。
- 同一回合对非只读工具使用规范化参数指纹去重，阻止模型在多轮中重复提交相同副作用动作。

## 第三阶段：结果约束

- `SkillDefinition` 使用 `side_effect` 分类；副作用操作仍按规范化参数去重，权限和管理员资格校验保持不变。
- 运行参数 `agent_max_skill_candidates` 控制每轮可见的候选工具数量。

## 验收与迁移

当前运行时可从 `AGENT_TURN_UPDATED` 事件得到候选工具、阶段轨迹、工具成功/拒绝/失败状态，用于计算工具成本、重复副作用率和任务完成率。固定回归场景覆盖：批量消息检索、工具权限过滤、异步参数校验、工具结果边界和重复副作用阻止。

迁移策略是默认启用候选检索，配置允许调整候选数量；没有引入第二个 Agent 进程或额外持久化服务。

## 已替代的后续阶段

原计划中的第二、三、四阶段已按现有队列和计划模式落地。语义向量检索、跨进程待确认持久化和自动化线上指标汇总不在当前单进程群聊框架的必要范围；现有事件轨迹为它们保留了可观测输入。

## 架构约束

- 工具元数据必须区分只读、幂等、可重试、副作用、确认和可见性；`retry_safe` 只是第一步。
- 模型只接收经过边界化的工具结果，不把任何工具/网页/记忆文本当作系统指令。
- 人格是稳定 system 身份，用户消息只承载用户内容与明确标注的参考上下文。
- 计划、执行和公开回复分离；没有可验证结果时不得宣称任务完成。
- 任何新 Agent 状态机先以可观测事件和测试落地，再接入真实群聊流量。
